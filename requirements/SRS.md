# Project Summary

**Project Name:** Andex AI — GitHub for Engineering Decisions

**Purpose:** Version-control engineering decisions and institutional knowledge so software teams keep a shared mental model as projects evolve.

**Problem Statement:** Engineering teams suffer **Mental Model Drift** — shared understanding of a project degrades as members change it rapidly with different tools, prompts, and schedules. Code is version-controlled; the decisions and knowledge behind it are not.

**Business Goal:** Provide human-in-the-loop, AI-assisted governance of engineering decisions: every change is proposed, analysed, voted on by the team, decided by a manager, tracked on a decision branch, and archived — with drift between the agreed model and reality detected automatically.

**Intended Users:** Software engineering teams — managers (project owners/deciders) and workers (engineers who suggest and vote).

**Stakeholders:** Engineering managers, engineers, product stakeholders who supply user feedback, organisational leadership (governed-memory consumers), hackathon sponsors (Conduct, CoralOS, Fetch.ai, BasedAI) as evaluation stakeholders.

**Project Scope:**
- Project ("Main Idea") creation from an uploaded brief, including parsed functional/non-functional requirements.
- Suggestion (proposal) lifecycle with AI-generated context, impact analysis, and review.
- Team voting and consensus tallying; manager-only accept/decline.
- Decision branches: creation on acceptance, implementation-task generation, merge-to-main, discard.
- Mental-model drift detection.
- Product discovery: clustering user feedback into ranked Feature Packs, promotable to suggestions.
- Enterprise memory governance: permission-filtered AI retrieval, lineage inheritance, temporal locks, audit logging.
- Transparent agent activity logging and skill catalogue.
- AI assistance through interchangeable LLM providers with graceful degradation.

**Out of Scope:**
- Writing, generating, or hosting source code (not an AI coding assistant).
- Code repository hosting (not a GitHub clone).
- Automatic document generation from code (not a documentation generator).
- Autonomous AI approval of any decision (humans always decide).
- Real external notification delivery (Slack/Discord/email are recorded as intents only).

# Domain Model

| Entity | Description | Relationships |
|---|---|---|
| Project (Main Idea / Project Brain) | The versioned, authoritative statement of a project: vision, goals, requirements, architecture components, institutional memory. | has many Requirements, Architecture Components, Memory Entries, Suggestions, Decision Branches |
| Requirement | A functional or non-functional requirement statement captured on the Project from a parsed brief or manager input. | belongs to Project |
| Architecture Component | A named element of the project's agreed structure (service, module, api, database, integration) with dependencies. | belongs to Project; depends on other Components |
| Institutional Memory Entry | A recorded decision or note that explains why the project is the way it is. | belongs to Project; may reference a Decision Branch |
| Team Member | A person with a workflow role (manager or worker) and a memory-governance role (manager, developer, intern). | authors Suggestions; casts Votes; makes Manager Decisions |
| Suggestion (Proposal) | A proposed change targeting the Main Idea or an open Decision Branch, enriched by AI analysis. | authored by Team Member; has Context, Impact Analysis, Review, Votes; may seed or extend a Decision Branch |
| Context | Related prior decisions, related architecture, and detected duplicates for a Suggestion. | belongs to Suggestion |
| Impact Analysis | Dependency/architecture/API/task impact items, cost estimate, and risk level for a Suggestion. | belongs to Suggestion |
| Review | AI-generated pros, cons, risks, trade-offs, open questions, suggested reviewers, and a team summary. | belongs to Suggestion |
| Vote | One team member's stance on a Suggestion (approve, reject, needs discussion, approve with comments) with optional comment. | belongs to Suggestion; cast by Team Member |
| Manager Decision | The manager's accept/decline outcome and note on a tallied Suggestion. | belongs to Suggestion; made by Manager |
| Decision Branch | A versioned line of accepted change: the Main Idea plus all accepted Suggestions on the branch, with lifecycle open → implementing → merged-to-main / discarded. | belongs to Project; seeded by a Suggestion; accumulates accepted Suggestions; has Implementation Tasks |
| Implementation Task | A unit of implementation work generated when a branch enters implementation. | belongs to Decision Branch; may reference a Suggestion |
| Drift Alert | A detected discrepancy between the agreed model and implementation or team behaviour. | relates to Project, Tasks, Suggestions |
| Agent Log | An auditable record of a single AI-agent action (input, output, timestamp). | may reference a Suggestion |
| Agent Skill | A declared capability of a specialist agent, published in a catalogue. | belongs to an Agent |
| Feedback Item | A single piece of user feedback (review, ticket, survey, social, analytics, interview) with sentiment, segment, geography. | may belong to Project; clustered into Feature Packs |
| Feature Pack | A discovered, evidence-backed feature theme with priority, confidence, impact, and status (discovered, promoted, dismissed). | derived from Feedback Items; may be promoted to a Suggestion |
| Governed Memory Record | A knowledge resource wrapped with permission metadata (visibility, allowed roles, temporal lock) and lineage. | derived from Project artefacts; parent/child lineage to other Records |
| Permission Audit Log | An immutable record of one access decision (who, what, action, granted, reason). | references Governed Memory Record and Team Member |

# Actors

**Manager**
- Description: Project owner and final decision authority (demo: one manager in a seven-person team).
- Responsibilities: create and update Projects from briefs; review tallied Suggestions; accept or decline with a note; start branch implementation; merge branches to main; discard branches.
- Business Permissions: create/update Project; accept/decline Suggestions; manage branch lifecycle; full read access to governed memory.

**Worker — Developer**
- Description: Engineer contributing suggestions and votes.
- Responsibilities: submit Suggestions to main or branches; vote on Suggestions; view analyses, branches, tasks, drift.
- Business Permissions: create Suggestions; one revisable Vote per Suggestion; read public and internal governed memory (confidential/leadership only when temporally unlocked); cannot make manager decisions.

**Worker — Intern**
- Description: Junior team member with the same workflow rights as a developer but minimal knowledge access.
- Responsibilities: as Worker — Developer.
- Business Permissions: as Worker — Developer for workflow; read **public** governed memory only.

**AI Specialist Agents (system actors)**
- Description: Ten specialist agents (Project Brain, Proposal, Product Discovery, Impact, Review, Consensus, Branch, Implementation, Communication, Drift Detection) that analyse and recommend.
- Responsibilities: enrich Suggestions, cluster feedback, tally support, generate tasks, detect drift, record notification intents.
- Business Permissions: may recommend and annotate; may never approve, decline, merge, or discard; may only receive knowledge that passed deterministic permission filtering.

**Feedback Provider (external)**
- Description: End users whose feedback enters product discovery.
- Responsibilities: none within the system (source of Feedback Items).
- Business Permissions: none.

# Functional Requirements

### FR-001 · Create Project from Brief
- **Description:** The system shall allow a manager to create a Project by uploading a project brief document.
- **Business Value:** Establishes the single source of truth the whole workflow governs.
- **Priority:** High · **Dependencies:** None · **Source:** src/app/api/project/parse/route.ts, src/lib/parse-project-brief.ts
- **User Story:** As a Manager, I want to create a project from a brief, so that my team has one agreed starting point.
- **Acceptance Criteria:**
  - A manager can submit a brief document and a project is created from it.
  - A non-manager attempting creation is refused.
  - The created project records its creator and an initial version.
- **Traceability Tags:** Project, Brief, Governance

### FR-002 · Parse Brief into Structured Fields
- **Description:** The system shall extract name, vision, goals, functional requirements, and non-functional requirements from an uploaded brief in structured or free-text form.
- **Business Value:** Turns unstructured intent into governable, reviewable content.
- **Priority:** High · **Dependencies:** FR-001 · **Source:** src/lib/parse-project-brief.ts
- **User Story:** As a Manager, I want my brief parsed into structured fields, so that requirements are individually visible and trackable.
- **Acceptance Criteria:**
  - Structured briefs populate all recognised fields.
  - Free-text briefs yield at minimum a name and vision when derivable.
  - Unrecognised sections are omitted rather than invented.
- **Traceability Tags:** Project, Requirements, Import

### FR-003 · View Project (Main Idea)
- **Description:** The system shall present each Project's vision, goals, requirements, architecture, institutional memory, and current version to all team members.
- **Business Value:** Shared visibility is the core defence against mental-model drift.
- **Priority:** High · **Dependencies:** FR-001 · **Source:** src/app/brain, src/app/api/project
- **User Story:** As a Worker, I want to view the main idea, so that I understand the agreed direction before suggesting changes.
- **Acceptance Criteria:**
  - Every team member can view every Project's current state and version.
  - Requirements captured at creation are listed individually.
- **Traceability Tags:** Project, Transparency

### FR-004 · Update Project
- **Description:** The system shall allow a manager to update a Project's editable fields.
- **Business Value:** Keeps the source of truth correct without bypassing decision governance for structural change.
- **Priority:** Medium · **Dependencies:** FR-001 · **Source:** src/app/api/project (PATCH)
- **User Story:** As a Manager, I want to correct project details, so that the main idea stays accurate.
- **Acceptance Criteria:**
  - A manager can update the project; a worker cannot.
  - Updates change the project's last-updated time.
- **Traceability Tags:** Project, Governance

### FR-005 · View Project Requirements
- **Description:** The system shall present the functional and non-functional requirements of each Project as a distinct view.
- **Business Value:** Requirements visibility enables impact analysis and downstream planning.
- **Priority:** Medium · **Dependencies:** FR-002 · **Source:** src/app/requirements/page.tsx
- **User Story:** As a Worker, I want to browse project requirements, so that my suggestions align with them.
- **Acceptance Criteria:**
  - Functional and non-functional requirements are listed separately per project.
  - Projects without captured requirements present an empty state, not an error.
- **Traceability Tags:** Requirements, Transparency

### FR-006 · Submit Suggestion
- **Description:** The system shall allow any team member to submit a Suggestion with title and description, targeting either a Project's main idea or a specific open Decision Branch.
- **Business Value:** Low-friction capture of proposed change is the entry point of governance.
- **Priority:** High · **Dependencies:** FR-001 · **Source:** src/lib/agents/orchestrator.ts (createProposal), src/app/api/proposals
- **User Story:** As a Worker, I want to submit a suggestion to main or to a branch, so that my idea enters the review workflow.
- **Acceptance Criteria:**
  - A suggestion requires a title and description and receives a unique identifier.
  - The author and target (main idea or branch) are recorded.
  - The suggestion enters the analysis workflow immediately.
- **Traceability Tags:** Suggestion, Workflow

### FR-007 · Gather Suggestion Context
- **Description:** The system shall automatically attach to each new Suggestion the related prior decisions, related architecture components, and a summary of what was retrieved.
- **Business Value:** Reviewers decide with institutional memory in view, not from scratch.
- **Priority:** High · **Dependencies:** FR-006 · **Source:** orchestrator (gatherContext)
- **User Story:** As a Worker, I want each suggestion enriched with context, so that voting is informed by history.
- **Acceptance Criteria:**
  - Context lists related prior decisions and related architecture components.
  - Context names the target (main idea or branch) it was gathered against.
- **Traceability Tags:** Suggestion, Context, InstitutionalMemory

### FR-008 · Detect Duplicate Suggestions
- **Description:** The system shall flag likely duplicates of a new Suggestion among existing non-rejected Suggestions and record them in its context.
- **Business Value:** Prevents parallel teams re-deciding the same question.
- **Priority:** Medium · **Dependencies:** FR-007 · **Source:** orchestrator (duplicates), fixtures edge case C1
- **User Story:** As a Manager, I want duplicates flagged, so that overlapping proposals are consolidated instead of decided twice.
- **Acceptance Criteria:**
  - A suggestion whose title substantially overlaps an existing active suggestion lists that suggestion as a duplicate.
  - Rejected suggestions are not reported as duplicates.
- **Traceability Tags:** Suggestion, Duplicates

### FR-009 · Analyse Suggestion Impact
- **Description:** The system shall produce an impact analysis for each Suggestion covering dependency, architecture, API, and task impact, with a cost estimate and an overall risk level.
- **Business Value:** Makes blast radius visible before anyone votes.
- **Priority:** High · **Dependencies:** FR-006 · **Source:** orchestrator (analyzeImpact)
- **User Story:** As a Worker, I want to see what a suggestion touches, so that I can vote on evidence.
- **Acceptance Criteria:**
  - Impact items identify affected targets with severities.
  - A cost estimate and a low/medium/high risk level are always present.
- **Traceability Tags:** Suggestion, Impact, Risk

### FR-010 · Generate Suggestion Review
- **Description:** The system shall generate for each Suggestion a review containing pros, cons, risks, trade-offs, open questions, suggested reviewers, and a team-visible summary.
- **Business Value:** Balanced framing raises decision quality and speed.
- **Priority:** High · **Dependencies:** FR-006 · **Source:** orchestrator (generateReview), ReviewAnalysis type
- **User Story:** As a Worker, I want a balanced review of each suggestion, so that discussion starts from both sides.
- **Acceptance Criteria:**
  - Every analysed suggestion has at least one pro, one con, and one open question.
  - A team summary is present and visible to all team members.
- **Traceability Tags:** Suggestion, Review, AIAssistance

### FR-011 · Vote on Suggestions
- **Description:** The system shall allow workers to cast a vote (approve, reject, needs discussion, approve with comments) with an optional comment on a Suggestion open for review.
- **Business Value:** Team consensus is captured explicitly rather than assumed.
- **Priority:** High · **Dependencies:** FR-006 · **Source:** orchestrator (castVote), src/lib/auth/team.ts (canVote)
- **User Story:** As a Worker, I want to vote on suggestions, so that team sentiment shapes the decision.
- **Acceptance Criteria:**
  - A worker can vote once per suggestion; re-voting replaces their previous vote.
  - A manager cannot cast votes.
  - Votes are refused on suggestions not open for review.
- **Traceability Tags:** Vote, Consensus, HumanInTheLoop

### FR-012 · Tally Votes
- **Description:** The system shall tally a Suggestion's votes and set it to "ready for manager" unless rejections outnumber approvals, in which case it becomes "needs discussion".
- **Business Value:** Routes contested items to discussion instead of a premature decision.
- **Priority:** High · **Dependencies:** FR-011 · **Source:** orchestrator (tallyVotes)
- **User Story:** As a Manager, I want contested suggestions separated from supported ones, so that my queue reflects team consensus.
- **Acceptance Criteria:**
  - More rejections than approvals yields "needs discussion".
  - Otherwise the suggestion becomes "ready for manager".
  - Tallying a suggestion with no votes is refused.
- **Traceability Tags:** Consensus, Workflow

### FR-013 · Manager Accepts Suggestion
- **Description:** The system shall allow only a manager to accept a tallied Suggestion, recording the decision and an optional note.
- **Business Value:** Preserves human accountability for every change to the agreed model.
- **Priority:** High · **Dependencies:** FR-012 · **Source:** orchestrator (managerAcceptProposal)
- **User Story:** As a Manager, I want to accept suggestions the team supports, so that agreed change moves forward under my accountability.
- **Acceptance Criteria:**
  - Acceptance by a non-manager is refused.
  - The decision, decider, and note are recorded on the suggestion.
  - Acceptance triggers branch handling per FR-015/FR-016.
- **Traceability Tags:** ManagerDecision, Governance, HumanInTheLoop

### FR-014 · Manager Declines Suggestion
- **Description:** The system shall allow only a manager to decline a tallied Suggestion with an optional explanatory note, closing it without changing any Project or Branch.
- **Business Value:** Explicit, explained rejection preserves trust and institutional memory.
- **Priority:** High · **Dependencies:** FR-012 · **Source:** orchestrator (managerDeclineProposal)
- **User Story:** As a Manager, I want to decline suggestions with a reason, so that authors understand the outcome.
- **Acceptance Criteria:**
  - Declining is manager-only and marks the suggestion rejected with the note retained.
  - No project or branch content changes as a result of a decline.
- **Traceability Tags:** ManagerDecision, Governance

### FR-015 · Create Decision Branch on Acceptance
- **Description:** When a manager accepts a Suggestion targeting the main idea, the system shall create a Decision Branch containing the main idea merged with the accepted Suggestion, leaving the main idea unchanged.
- **Business Value:** Changes accumulate safely on branches; main remains the stable agreed truth until merge.
- **Priority:** High · **Dependencies:** FR-013 · **Source:** orchestrator (createDecisionBranch/mergeProposalIntoBrain)
- **User Story:** As a Manager, I want accepted ideas isolated on a branch, so that main only changes when I merge.
- **Acceptance Criteria:**
  - Acceptance of a main-targeting suggestion creates a branch seeded by that suggestion, with an incremented version.
  - The branch records the main version at creation; the main idea itself is unmodified.
- **Traceability Tags:** Branch, Versioning

### FR-016 · Extend Branch with Accepted Suggestion
- **Description:** When a manager accepts a Suggestion targeting an existing open Decision Branch, the system shall merge the Suggestion into that branch's content and increment the branch version.
- **Business Value:** Related decisions compose on one branch instead of fragmenting.
- **Priority:** High · **Dependencies:** FR-013, FR-015 · **Source:** orchestrator (managerAcceptProposal branch path)
- **User Story:** As a Worker, I want to suggest additions to an in-flight branch, so that a decision evolves coherently.
- **Acceptance Criteria:**
  - The branch's content and version are updated and the accepted suggestion is recorded on the branch.
  - The main idea remains unchanged.
- **Traceability Tags:** Branch, Suggestion, Versioning

### FR-017 · Start Branch Implementation
- **Description:** The system shall allow a manager to move an open Decision Branch to "implementing", generating implementation tasks for the branch's accepted change.
- **Business Value:** Connects approved decisions to actionable work.
- **Priority:** High · **Dependencies:** FR-015 · **Source:** orchestrator (startBranchImplementation), branches route
- **User Story:** As a Manager, I want to start implementation of a branch, so that the team receives concrete tasks.
- **Acceptance Criteria:**
  - Starting implementation is manager-only.
  - Tasks are generated and linked to the branch; the branch status becomes "implementing".
- **Traceability Tags:** Branch, Task, Workflow

### FR-018 · Merge Branch to Main
- **Description:** The system shall allow a manager to merge an open or implementing Decision Branch to main, replacing the Project's content with the branch content and adopting its version.
- **Business Value:** The single moment the agreed truth changes — explicit, versioned, attributable.
- **Priority:** High · **Dependencies:** FR-015 · **Source:** orchestrator (mergeBranchToMain)
- **User Story:** As a Manager, I want to merge a finished branch, so that the main idea reflects the accepted decisions.
- **Acceptance Criteria:**
  - Merging is manager-only and only from open or implementing status.
  - After merge, the project's version equals the branch version and the branch is marked merged with a timestamp.
- **Traceability Tags:** Branch, Versioning, Governance

### FR-019 · Discard Branch
- **Description:** The system shall allow a manager to discard an open or implementing Decision Branch, closing it without any change to the main idea.
- **Business Value:** Reversibility — abandoned directions leave an auditable trace but no residue.
- **Priority:** High · **Dependencies:** FR-015 · **Source:** orchestrator (discardBranch)
- **User Story:** As a Manager, I want to discard a branch that didn't work out, so that main is untouched and the record remains.
- **Acceptance Criteria:**
  - Discarding is manager-only and only from open or implementing status.
  - The project's version and content are unchanged; the branch is marked discarded with a timestamp.
- **Traceability Tags:** Branch, Reversibility

### FR-020 · View Branch History
- **Description:** The system shall present all Decision Branches per Project with status, versions, seed suggestion, and accepted suggestions.
- **Business Value:** The decision history is the "commit log" of the project's evolution.
- **Priority:** Medium · **Dependencies:** FR-015 · **Source:** src/app/branches, branches route
- **User Story:** As a Worker, I want to browse branch history, so that I can see how and why the project changed.
- **Acceptance Criteria:**
  - Branches list their status (open, implementing, merged to main, discarded) and lifecycle timestamps.
- **Traceability Tags:** Branch, Transparency, Versioning

### FR-021 · View Implementation Tasks
- **Description:** The system shall present implementation tasks with status, branch linkage, and affected components.
- **Business Value:** Makes decision-to-work traceability visible.
- **Priority:** Medium · **Dependencies:** FR-017 · **Source:** src/app/tasks, tasks route
- **User Story:** As a Worker, I want to see the tasks a decision produced, so that I know what to build.
- **Acceptance Criteria:**
  - Every task shows its status and the branch it belongs to.
- **Traceability Tags:** Task, Transparency

### FR-022 · Detect Mental-Model Drift
- **Description:** The system shall scan for drift and raise alerts when implementation tasks reference components absent from the agreed model, or when the count of open suggestions indicates diverging priorities.
- **Business Value:** The product's core promise — divergence is surfaced instead of silently accumulating.
- **Priority:** High · **Dependencies:** FR-003, FR-021 · **Source:** orchestrator (detectDrift), docs/TESTING.md
- **User Story:** As a Manager, I want drift alerts, so that I can realign the team before divergence hardens.
- **Acceptance Criteria:**
  - A pending task referencing a component not in the project model produces an alert with a recommendation.
  - An excessive open-suggestion backlog produces an alert.
  - Alerts carry severity and detection time and are viewable on demand.
- **Traceability Tags:** Drift, InstitutionalMemory

### FR-023 · Log Agent Activity
- **Description:** The system shall record every AI-agent action as an immutable log entry (agent, action, input, output, related suggestion, timestamp) and present the log to all team members.
- **Business Value:** Auditable AI — every recommendation is traceable to its origin.
- **Priority:** High · **Dependencies:** None · **Source:** orchestrator (log), src/app/agents
- **User Story:** As a Manager, I want a full agent audit trail, so that AI involvement is transparent and reviewable.
- **Acceptance Criteria:**
  - Every pipeline step writes exactly one log entry.
  - The log is viewable, newest first, by all team members.
- **Traceability Tags:** AuditLog, Transparency, AIExplainability

### FR-024 · Publish Agent Skill Catalogue
- **Description:** The system shall publish a catalogue of each specialist agent's declared skills.
- **Business Value:** Team members and integrators can see exactly what each agent can and cannot do.
- **Priority:** Low · **Dependencies:** None · **Source:** src/lib/agents/skills.ts, agents route
- **User Story:** As a Worker, I want to see each agent's skills, so that I understand the automation acting on our decisions.
- **Acceptance Criteria:**
  - Every specialist agent lists at least one named skill in the catalogue.
- **Traceability Tags:** AIExplainability, Transparency

### FR-025 · Record Notification Intents
- **Description:** The system shall record a notification intent (channel and message) at key workflow moments, without asserting external delivery.
- **Business Value:** Communication trail exists now; delivery integrations can attach later.
- **Priority:** Low · **Dependencies:** FR-006 · **Source:** orchestrator (sendNotification)
- **User Story:** As a Manager, I want workflow events to produce notification records, so that the communication trail is complete.
- **Acceptance Criteria:**
  - Suggestion submission and branch merge each record a notification intent naming the channel.
- **Traceability Tags:** Notification, AuditLog

### FR-026 · Ingest User Feedback
- **Description:** The system shall accept user feedback items (source, text, sentiment, user segment, optional geography and project) for product discovery.
- **Business Value:** Grounds discovery in real user evidence.
- **Priority:** Medium · **Dependencies:** None · **Source:** src/lib/discovery/mock-feedback.ts, discovery route
- **User Story:** As a Manager, I want user feedback in the system, so that discovery works from evidence, not intuition.
- **Acceptance Criteria:**
  - Feedback items retain source, sentiment, segment, and timestamp.
- **Traceability Tags:** Discovery, Feedback, Import

### FR-027 · Generate Ranked Feature Packs
- **Description:** The system shall cluster related feedback into Feature Packs, each with a user problem, suggested feature, evidence quotes, affected segments, geographic insights, pros/cons/risks, and priority, confidence, and impact ratings.
- **Business Value:** Converts raw feedback into decision-ready, evidence-backed candidates.
- **Priority:** Medium · **Dependencies:** FR-026 · **Source:** src/lib/agents/product-discovery.ts
- **User Story:** As a Manager, I want feedback distilled into ranked feature packs, so that I can weigh opportunities quickly.
- **Acceptance Criteria:**
  - Each pack cites its supporting evidence count and quotes.
  - Each pack carries priority and confidence scores and an impact rating.
  - Packs are ordered by priority.
- **Traceability Tags:** Discovery, FeaturePack, AIAssistance

### FR-028 · Promote or Dismiss Feature Packs
- **Description:** The system shall allow a team member to promote a discovered Feature Pack into a Suggestion (entering the standard workflow) or dismiss it; either way the pack records its outcome.
- **Business Value:** Discovery feeds governance — with a human deciding what enters the pipeline.
- **Priority:** Medium · **Dependencies:** FR-027, FR-006 · **Source:** product-discovery (featurePackToProposalText), discovery route, feature-packs page
- **User Story:** As a Manager, I want to promote a feature pack to a suggestion, so that evidence-backed ideas get formally decided.
- **Acceptance Criteria:**
  - Promotion creates a suggestion carrying the pack's evidence and links the pack to it.
  - Dismissal marks the pack dismissed without creating a suggestion.
  - A pack is never promoted automatically.
- **Traceability Tags:** Discovery, Suggestion, HumanInTheLoop

### FR-029 · Register Governed Memory with Permissions
- **Description:** The system shall wrap knowledge resources (project brains, memory entries, feature packs, proposals, AI summaries, and similar) as governed memory records carrying owner, visibility (public, internal, confidential, leadership), allowed roles, optional user allow-list, optional unlock time, and lineage.
- **Business Value:** Organisational knowledge becomes access-controlled data rather than an undifferentiated blob.
- **Priority:** High · **Dependencies:** FR-003 · **Source:** src/lib/governance/memory-registry.ts, types (GovernedMemoryRecord)
- **User Story:** As a Manager, I want knowledge classified with permissions, so that sensitive content is structurally protected.
- **Acceptance Criteria:**
  - Every governed record carries visibility, allowed roles, owner, and lineage metadata.
- **Traceability Tags:** MemoryGovernance, Permissions

### FR-030 · Enforce Deterministic Permission Filtering Before AI Retrieval
- **Description:** The system shall evaluate access to governed memory with deterministic rules (role, visibility, allow-list, temporal lock, lineage) and shall pass only authorised records to any AI model during retrieval or question answering.
- **Business Value:** Permission enforcement cannot be prompt-injected or hallucinated away — it happens before the model sees anything.
- **Priority:** High · **Dependencies:** FR-029 · **Source:** src/lib/governance/permission-service.ts (canAccess), memory-retrieval.ts (askWithGovernedMemory)
- **User Story:** As a Manager, I want restricted knowledge filtered before AI processing, so that the model can never leak what it never received.
- **Acceptance Criteria:**
  - Access decisions derive solely from permission metadata, never from AI judgement.
  - An intern's query retrieves public records only; a developer's adds internal; a manager's retrieves all.
  - Unauthorised records are absent from AI context, and the response reports how many were filtered.
- **Traceability Tags:** MemoryGovernance, Permissions, AIExplainability, Security

### FR-031 · Inherit Restrictions Through Lineage
- **Description:** The system shall give derived memory (summaries, embeddings, promoted content) an effective permission no weaker than its source: the most restrictive visibility, the intersection of allowed roles, and the latest unlock time along its lineage.
- **Business Value:** Summarising a confidential document can never declassify it.
- **Priority:** High · **Dependencies:** FR-029 · **Source:** permission-service.ts (resolveEffectivePermissions)
- **User Story:** As a Manager, I want derived artefacts to inherit source restrictions, so that derivation is never an access loophole.
- **Acceptance Criteria:**
  - A record derived from a confidential parent is at least confidential regardless of its own declared visibility.
  - Role access to a derived record is the intersection of its own and its ancestors' allowed roles.
- **Traceability Tags:** MemoryGovernance, Lineage, Security

### FR-032 · Enforce Temporal Locks
- **Description:** The system shall deny access to non-public governed memory before its unlock time, and permit role-appropriate access after it.
- **Business Value:** Supports embargoed decisions (reorgs, strategy) that must stay sealed until a date.
- **Priority:** Medium · **Dependencies:** FR-029 · **Source:** permission-service.ts (isUnlocked)
- **User Story:** As a Manager, I want embargoed content locked until its date, so that timing-sensitive knowledge is protected.

# Business Rules

- **BR-001** — Only a manager may create or update a Project, accept or decline Suggestions, or change branch lifecycle state (start implementation, merge to main, discard).
- **BR-002** — Only workers vote; a manager never casts votes.
- **BR-003** — AI agents recommend; humans decide. No AI action may accept, decline, merge, discard, or auto-approve anything.
- **BR-004** — Vote tallying never approves: it routes to "ready for manager" or "needs discussion" (when rejections outnumber approvals) only.
- **BR-005** — The main idea is never modified directly by a suggestion; accepted change lives on a Decision Branch until an explicit merge to main.
- **BR-006** — Only merge-to-main changes a Project's content and version; a discarded branch leaves main untouched.
- **BR-007** — Branch lifecycle transitions (merge, discard) are permitted only from "open" or "implementing" status.
- **BR-008** — Each team member holds at most one vote per suggestion; a new vote replaces the old.
- **BR-009** — Governed-memory access decisions are deterministic (metadata-based); AI judgement never grants access.
- **BR-010** — Derived memory inherits the most restrictive permissions along its lineage (visibility maximum, role intersection, latest unlock time).
- **BR-011** — Non-public memory under a temporal lock is inaccessible to non-managers until its unlock time.
- **BR-012** — Memory roles scope reading: intern → public; developer → public + internal (confidential/leadership only when unlocked); manager → all.
- **BR-013** — Every permission decision, granted or denied, is audit-logged with a reason.
- **BR-014** — Feature Packs become Suggestions only through explicit human promotion.
- **BR-015** — Every AI-agent action is logged; the log is visible to the whole team.

# Assumptions

- **AS-001** — The seven-member demo team (one manager, six workers) stands in for real organisational identity; production deployments will integrate an identity provider.
- **AS-002** — Notification delivery (Slack/Discord/email/GitHub) is out of scope for this phase; recording intent is sufficient.
- **AS-003** — User feedback arrives pre-collected (mock dataset in this phase); live ingestion integrations come later.
- **AS-004** — A single organisation per deployment is assumed; the permission metadata carries an organisation identifier for future multi-tenancy.
- **AS-005** — Vote weighting is uniform; no quorum size is enforced beyond "at least one vote to tally".
- **AS-006** — English is the working language of briefs and suggestions; the parser has English section-header heuristics.
- **AS-007** — The persistence layer is assumed available; when it is not, the system operates for demo purposes with in-memory state.

# Constraints

- **CON-001 (Business)** — Human-in-the-loop is non-negotiable: no autonomous AI approval, in any configuration (hackathon judging criterion and product identity).
- **CON-002 (Business)** — Deliverable within a 7-day hackathon by a 7-person student team; scope must demo end-to-end.
- **CON-003 (Operational)** — Must run on free-tier AI providers with their rate limits; provider fallback exists partly for this reason.
- **CON-004 (Legal)** — Institutional memory may contain personal data; GDPR-relevant handling (visibility classes, audit logs) is required for EU deployments.
- **CON-005 (Technical, explicitly stated)** — Web application deployed on Vercel; persistence on MongoDB Atlas; AI via OpenAI-compatible provider APIs (README Tech Stack).

# Risks

- **RSK-001 (AI)** — Heuristic/AI-generated impact and review content may be generic or wrong, eroding trust in analyses. Mitigation: explainability (NFR-012), human decision points (BR-003).
- **RSK-002 (Security)** — Demo identity switching means any user can act as the manager; real deployments without authentication would void all governance guarantees (see CLR-001).
- **RSK-003 (AI/Privacy)** — If any retrieval path bypasses the deterministic filter, restricted memory could reach a model. Mitigation: single retrieval entry point (FR-030), audit logging (FR-033).
- **RSK-004 (Business)** — Voting-then-manager-decision could be perceived as decision theatre if managers routinely override consensus; adoption depends on trust in the workflow.
- **RSK-005 (Operational)** — Free-tier provider rate limits can throttle demos at the worst moment. Mitigation: multi-provider fallback (FR-036).
- **RSK-006 (Technical)** — Fallback in-memory state (AS-007) silently loses data on restart if operators mistake it for persistence.
- **RSK-007 (Adoption)** — The workflow adds process overhead per change; small teams may bypass it, recreating drift.
- **RSK-008 (AI)** — Duplicate detection is title-similarity-based and may miss semantic duplicates or flag false positives.

# Acceptance Summary

A manager founds a project by uploading a brief, which is parsed into a versioned "main idea" with visible requirements. Team members submit suggestions against main or an open branch; each suggestion is automatically enriched with historical context, duplicate flags, impact analysis, and a balanced review. Workers vote; tallying routes contested items to discussion and supported items to the manager, who alone accepts (spawning or extending a decision branch) or declines (with a note). Branches move through implementation (generating tasks) and end merged to main — the only event that changes the agreed truth and its version — or discarded without residue. Drift between the agreed model and actual work is detected and surfaced. In parallel, user feedback is clustered into evidence-backed, ranked feature packs that humans may promote into the same governed pipeline. All institutional knowledge is wrapped in permission metadata; AI question-answering sees only deterministically authorised memory, restrictions inherit through derivation, embargoes hold until their date, and every access decision and agent action is audit-logged. AI assistance rides on interchangeable providers with fallback and degrades gracefully to a fully functional non-AI workflow.
